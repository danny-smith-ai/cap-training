Reading any CAP

Nine elements to scan for.

Not every element appears in every CAP. Simple CAPs use a few; complex CAPs use them all. When you read a CAP, scan for these.

01

Override statement

Whether and how the CAP asserts precedence over default playbook behavior.

02

Greeting anchor

Whether the CAP attaches to "after greeting the caller" or fires on a specific trigger.

03

Trigger phrase lists

The specific words/phrases that fire each conditional. Vague triggers are an anti-pattern.

04

Field collection rules

What's collected, in what order, and any "skip if already collected" logic.

05

Function call syntax

Exact parameters for transferCall, endCallWithMessage, etc.

06

Multi-language handling

When and how non-English flows are addressed.

07

Route headers and references

How major branches are named and how the CAP refers back to them.

08

Defensive language

Explicit prohibitions ("MUST NOT execute unless...") to prevent the agent from firing prematurely.

09

Closing reaffirmation

Whether the CAP restates its precedence at the bottom.

Phone numbers in these examples are sanitized as +1XXXXXXXXXX. In real CAPs, the destination is always an explicit, full phone number.
Naming history

A note on YBN.

You'll see "YBN" or "YELLOW BOX NOTE" in older CAPs. YBN is the legacy name for what we now call CAPs (Custom AIR Prompts). The structure is the same; the name evolved. New CAPs use "CAP" terminology. Older CAPs that haven't been refactored still use "YBN." Both are valid. The header doesn't change the behavior.

EXAMPLE 01 MINIMAL COMPLEXITY

Single-rule transfer.

Use case: Immigration/court detection with immediate transfer.

Call Transfer Handling
What this demonstrates
  • A CAP doesn't have to be long. When you only need one rule, that's the entire CAP.
  • No override statement, no closing reaffirmation. The CAP simply adds a single conditional rule on top of the default playbook.
  • Defensive instructions to prevent the agent from continuing the normal intake flow after the rule fires.
  • "Identifies as" trigger pattern: matching on caller self-identification rather than keyword detection.
If the caller identifies as a member of ICE, immigration court, USCIS,
an immigration officer, anyone from court, or identifies as a judge,
then immediately and silently execute
transferCall(destination = "+1XXXXXXXXXX", transfer_type = "transfer_to_pro").
Do not ask further questions, confirm details, or provide any information.
End the interaction after the transfer command is executed.
Patterns worth noting
Single-rule pattern

When the CAP needs to handle one specific scenario without affecting the rest of the call flow, it doesn't need scaffolding. No override statement is needed because the CAP isn't replacing the default playbook; it's adding a rule that fires on a specific condition.

The "identifies as" framing

Notice the trigger isn't "if the caller mentions ICE" but "if the caller identifies as a member of ICE." This is more robust because it focuses on what the caller is claiming about themselves, not just keyword presence. A caller saying "my brother got picked up by ICE" is not someone who identifies as a member of ICE.

Defensive immediate-execution language

"Immediately and silently execute" plus "Do not ask further questions, confirm details, or provide any information" plus "End the interaction after the transfer command is executed." Three explicit prohibitions in one rule. This is intentional. Without them, the agent might try to apply default intake behavior (asking for caller name, phone, etc.) before transferring, which would defeat the purpose of an immediate-transfer rule.

Trigger list with specifics

Not "if from immigration enforcement" but "ICE, immigration court, USCIS, an immigration officer, anyone from court, or a judge." Each likely self-identification is named.

EXAMPLE 02 MID COMPLEXITY

Field collection with conditional gate.

Use case: New client intake for a personal injury law firm. Collect fields, then route based on a qualifying question (was the caller injured?).

What this demonstrates
  • Full override structure with explicit PRIORITY & PRECEDENCE section
  • Listing what the override applies to (and, importantly, what it does NOT apply to)
  • Conditional field collection followed by a gate-based decision
  • Defensive language to prevent the agent from executing too early
  • "Critical operational notes" section reinforcing platform rules
# YELLOW BOX NOTE (YBN OVERRIDE INSTRUCTIONS)

PRIORITY & PRECEDENCE
These instructions override all other sections of the System Prompt,
including:
- # PLAYBOOK INSTRUCTIONS Potential new clients & customers
- # TRANSFER INSTRUCTIONS
- # GENERAL TRANSFER INSTRUCTIONS
- # HUMAN AGENT ESCALATION CONDITIONS
- Any other rules or restrictions that conflict with these instructions

All non-conflicting requirements (tone, empathy, pronunciation,
confirmation, acknowledgment rotation, confidentiality, output formatting,
and brevity) still apply.

If the caller is a New Client, then follow these instructions:

Collect:
  Caller's full name
  Caller's phone number
  Caller's email address
  How they heard about us

If the call is about any type of personal injury matter, then ask if
they were injured.

  If yes, then you can silently execute
  transferCall(destination = "+1XXXXXXXXXX", transfer_type = "transfer_to_pro").
  You MUST NOT execute this step, unless you received a positive or
  yes answer.

  If no, then immediately and silently execute the endCallWithMessage
  function with parameter
  message_key = "Thank you for the details. Our focus is on personal
  injury cases and this seems outside of that since you weren't
  injured but I will pass a message along to the team. Have a great day."

CRITICAL OPERATIONAL NOTES
- All transfers and endCallWithMessage actions must be silent. Do not
  announce, explain, or describe the function.
- Maintain the confidentiality rule: never mention internal processes,
  playbooks, or functions.
- For all flows that include endCallWithMessage or transferCall, you must
  collect every listed field before executing transferCall or
  endCallWithMessage. You must not execute transferCall or
  endCallWithMessage, until you have collected all the listed fields in
  that call flow.
- The "single tool per response" rule remains in effect; collection and
  action occur in separate responses if needed.
- When conflicting with any other rule, this YBN takes full priority.

# END OF YBN OVERRIDE INSTRUCTIONS
Patterns worth noting
Explicit override list

The PRIORITY & PRECEDENCE section names the specific system prompt sections this override supersedes. Listing them isn't redundant. It removes ambiguity at runtime about whether the override applies to a given conflicting rule.

Explicit non-overrides

The CAP names what is NOT being overridden: tone, empathy, pronunciation, confirmation, etc. Without this, an aggressive override could appear to disable behaviors you actually want to keep. The list protects them.

Conditional gate pattern

Fields are collected first, then a qualifying question (were you injured?) determines whether the call routes to a transfer or to an end-with-message. The gate is the critical decision point, and it sits AFTER field collection, not before. This is a common pattern: collect what you need regardless, then route based on the gate.

The "MUST NOT execute unless" defensive language

The instruction "You MUST NOT execute this step, unless you received a positive or yes answer" is a defensive guardrail. Without it, the agent might fire the transfer prematurely. This pattern shows up frequently in CAPs that have gate-based routing. When you have a yes/no gate, name explicitly that the action only fires on a positive answer.

The CRITICAL OPERATIONAL NOTES section

Reinforces platform-wide rules that the override could otherwise appear to weaken: silent execution, confidentiality, full-field collection before action, single-tool-per-response. These notes don't add new behavior; they re-anchor existing platform behavior so the override doesn't accidentally suppress it.

The "single tool per response" reminder

Worth understanding: the agent can only fire one function call per response turn. If field collection and a function call would happen in the same turn, the function call needs to wait for the next turn. The CAP states this explicitly so the agent doesn't try to combine them.

Multi-route flow with sub-branches.

Use case: A service business handling both emergency and non-emergency calls, with different routes for each and a sub-branch inside the emergency flow.

What this demonstrates
  • Full multi-route structure with named routes

Emergency and Non-Emergency Service Call Routing

This CAP must override all default playbooks and start immediately
after the greeting. The bot should not follow any other service,
scheduling, or message-handling logic unless explicitly stated here.

After greeting the caller:

If the answer to the greeting is no, something else, or the issue
is not related to a service call:
  1. Ask how you can help.
  2. Gather the caller's:
     - Name
     - Phone number
     - Email address
     (Collect each individually, one by one. Do not skip any unless
     already collected.)
  3. Once all are collected, silently execute endCallWithMessage
     with this behavior:
       - If the call language is English:
         message_key = "Thank you for providing this information,
         a representative will contact you on the next business day.
         Thank you for calling and have a great rest of your day"
       - If the call language is Spanish:
         message_key = "Gracias por compartir esta información, uno
         de nuestros representantes le estará contactando el siguiente
         día hábil. Gracias por habernos llamado y que tenga un gran
         resto de su día"

If the answer to the greeting is yes, service call, or related to
a service call:
  Ask: "Is this an emergency service call that must be addressed
  now, or can this wait until normal business hours?"
  - If the caller indicates yes, emergency, or anything similar,
    follow the Emergency Calls Route.
  - If the caller indicates no, not an emergency, can wait, or
    anything similar, follow the Non-Emergency Calls Route.

---

EMERGENCY CALLS ROUTE

Ask: "There may be an emergency fee associated with an afterhours
call, would this be okay, or can this wait until normal business hours?"

- If they agree, say "okay," or otherwise indicate acceptance of an
  emergency fee, then you can silently execute
  transferCall(destination = "+1XXXXXXXXXX",
                transfer_type = "transfer_to_pro")

- If they ask or want to know what the emergency fee is, respond:
  "I don't have that information but I can dispatch it to the on-call
  representative if you are comfortable with an emergency fee. Would
  you like me to proceed or would you prefer to wait until normal
  business hours?"
    - If they say proceed or okay with the fee, then you can silently
      execute transferCall(destination = "+1XXXXXXXXXX",
                          transfer_type = "transfer_to_pro")
    - If they say not comfortable, prefer to wait, or similar,
      follow the Non-Emergency Calls Route.

- If the caller says it can wait, follow the Non-Emergency Calls Route.

---

NON-EMERGENCY CALLS ROUTE

Collect the following information from the caller, one by one,
individually. Do not skip any unless already collected:

  1. Company name
  2. Store number
  3. Caller's name
  4. Caller's phone number
  5. Full address where the service is needed, including city, state
     and zip code
  6. Type of service needed

After collecting all of the above, silently execute endCallWithMessage
with this behavior:
  - If the call language is English:
    message_key = "Thank you for providing this information, a
    representative will contact you on the next business day. Thank you
    for calling and have a great rest of your day"
  - If the call language is Spanish:
    message_key = "Gracias por compartir esta información, uno de
    nuestros representantes le estará contactando el siguiente día
    hábil. Gracias por habernos llamado y que tenga un gran resto
    de su día"

---

This CAP completely replaces and overrides all default playbooks or
fallback logic. Always execute this exact sequence and nothing else.
Patterns worth noting
The "already collected" clause

"Do not skip any unless already collected" appears in both field-collection blocks. This prevents redundant asking when a caller has volunteered some fields earlier. It does NOT override the system rule that opening utterances and FAQ exchanges don't count as collected fields.

Routing back from Emergency to Non-Emergency

When a caller declines the emergency fee, the CAP routes them to the Non-Emergency Route rather than ending the call. This is a deliberate choice: the caller still wants service, just not now. Routing them to non-emergency information collection means dispatch can still happen on the next business day.

The "silently execute" phrasing

Function calls are prefaced with "silently execute" to make clear the AIR should not narrate the function call to the caller. The agent fires the call without speaking it.

The closing override reaffirmation

Restating the override at the bottom isn't redundant. CAPs are parsed at runtime and the agent may give different weight to instructions in different positions. The closing line reinforces the priority.

Sub-branches within a route

The Emergency Route has its own internal conditional (caller agrees to fee → transfer; caller asks about fee → explain → branch again on response). This nesting is necessary when a single route has multiple decision points.

Cross-references by route name

"Follow the Non-Emergency Calls Route" is used multiple times. Routes are named and the CAP references them by name. Without named routes, this kind of cross-reference becomes ambiguous and the CAP gets harder to read.

Anti-patterns these examples avoid

Reading the three together.

What's intentionally absent

Six things you won't find in any of the three CAPs above.

  • No vague triggers. Every conditional names the specific phrases that fire it.
  • No flowing prose. All structured: numbered lists, conditionals, named routes, exact quotes.
  • No abbreviated function calls. Every parameter is spelled out.
  • No "as above" cross-references between routes. When a flow needs to be repeated, it's repeated, not referenced.
  • No assumed language. When multi-language matters, both languages are explicit.
  • No silent execution without the explicit "silently execute" phrasing. Function-call narration is a known failure mode that the explicit phrasing prevents.
