The call lifecycle at a glance

Six stages, one shape.

Most calls follow this same arc: greeting, Main runs its checks, Main hands to Intake, Intake collects, end action fires, call ends. Variations layer on top.

Greeting

Caller dials, AIR answers and greets.

Main runs

Classifies caller, checks FAQ, transfer, and spam intent.

Handoff

Main hands off to Intake with the right playbook for the caller's type.

Intake runs

Walks the caller through playbook steps, collects fields, runs qualification.

End action

Transfer, end with message, or scheduling fires.

Call ends

The conversation closes.

Common variations

Where the standard path bends.

  • A Service-based qualification step may run before Intake to match the caller's need to a specific playbook
  • The FAQ Agent can be invoked at any point during the call to answer info-seeking questions
  • The Escalation Agent handles transfers, usually at the end of intake but can interrupt mid-flow if needed
  • The Lookup Agent runs during Intake when external data needs to be retrieved
A worked happy-path example

The simplest possible case.

Single-playbook account, no qualification steps, three mandatory fields (Full name, Phone, Email).

  1. SYSTEM
    Greeting plays
  2. CALLER
    "Hi, I need help with a contract review."
  3. MAIN
    STEP 1: descriptive context, skip FAQ
  4. MAIN
    STEP 2: no transfer intent, continue
  5. MAIN
    STEP 2.5: not spam/sales, continue
  6. MAIN
    STEP 3 (single-playbook accounts): silently calls getPlaybookInstructions with the only playbook UUID
  7. SYSTEM
    Intake Assistant loads
  8. INTAKE
    "I'd be happy to help. What's your full name?"
  9. CALLER
    "Jane Smith"
  10. INTAKE
    silently calls getOrCreateProfile(is_confirmed=false, first_name="Jane", last_name="Smith")
  11. INTAKE
    "Got it, and your phone number?"
  12. CALLER
    Caller provides phone
  13. INTAKE
    "And your email address?"
  14. CALLER
    Caller spells out email
  15. INTAKE
    repeats email back letter-by-letter for confirmation, no other question, no other function call
  16. CALLER
    Caller confirms
  17. INTAKE
    runs Completeness Audit, all 3 fields present
  18. INTAKE
    "Is there anything else I can help you with today?"
  19. CALLER
    "No, that's it."
  20. INTAKE
    silently calls endCallWithFinalStatement(playbook_id="...")
  21. SYSTEM
    Call ends

This is the simplest possible case: one playbook, no qualification, straight collection of three fields. Most accounts are more complex but follow the same shape.

A note on the function calls. Some of the calls shown above (getPlaybookInstructions, getOrCreateProfile) happen automatically as part of how the system runs. CAPs do not script those. The Callable Actions section below covers the actions a CAP actually scripts.

The agents

Seven agents. Three receive CAPs.

The Smith.ai system is multi-agent. A single call hands off between agents depending on what's needed. Each agent has its own instructions, its own data access, and its own actions it can take.

The single most important fact

Only Main, Intake, and Lookup receive SPECIAL INSTRUCTIONS (the CAP). The others are out of CAP reach.

Receives CAP · YES

Main Assistant

The entry-point. Greets the caller, classifies them, checks for FAQ-style questions, transfer requests, and spam/sales, then hands off to Intake with the right playbook.

Receives CAP · YES

Intake Assistant

The playbook executor. Walks the caller through the steps, collects required fields, runs qualification, and fires the end action.

Most CAPs target this agent.

Receives CAP · YES (under dedicated header)

Lookup Assistant

Handles external data lookups (typically backed by n8n in most accounts).

Native flow: understand, lookup, done.

Receives CAP · NO

FAQ Agent

Answers info-seeking questions from the dashboard FAQ list, Company Information, and Services list.

Receives CAP · NO

Scheduling Agent

Books, reschedules, and cancels appointments.

Receives CAP · NO

Escalation Agent

Executes transfers. Has access to the Team Directory, which Intake does not.

Receives CAP · NO

Service Type Agent

Handles service-based qualification (matches caller need to the account's service list).

What a CAP looks like

Plain structured language.

CAPs are written in plain structured language. Always start simple. Complexity only when needed.

Do

Conventions to follow.

  • Override statement at the top if the CAP supersedes default playbooks
  • Headers for routes/sections (Emergency Route, Non-Emergency Route, etc.)
  • Conditional logic in plain "If / If" blocks, with clear trigger phrases listed
  • Function calls quoted exactly with backticks, including all parameters
  • Numbered lists for sequenced steps
  • Each field on its own line when collecting multiple
  • Multi-language conditions stated explicitly
Don't

Anti-patterns.

  • Writing CAPs as flowing prose paragraphs
  • Vague triggers ("if the caller seems frustrated")
  • Function calls without exact parameter values
  • Missing the override statement when one is needed
  • Reusing rules across routes without explicit reference (copy them, do not reference "as above")
Skeleton structure
[CAP Name]: [What it does]

[Override / scope statement, if applicable]

After greeting the caller:

If [condition A]:
  1. [Action]
  2. [Action]
  3. Silently execute [function call with exact parameters]

If [condition B]:
  Ask: "[exact phrasing]"
  - If [response type 1], follow [Route X]
  - If [response type 2], follow [Route Y]

### ROUTE X
  [Steps for this route]

### ROUTE Y
  [Steps for this route]

The structure matters because the agent parses these rules at runtime. Ambiguity in language becomes ambiguity in behavior. Long narrative paragraphs are an anti-pattern. Structured, declarative blocks are the right shape.
