01

Core terms

The foundational vocabulary. Start here.

AIRAI Receptionist

Smith.ai's AI-powered call answering product. The AIR handles inbound calls for client accounts: greets callers, asks qualifying questions, collects information, schedules appointments, transfers calls, and ends calls. Each client account has its own configured AIR.

CAPCustom AIR Prompt

Custom prompt instructions inserted into the SPECIAL INSTRUCTIONS field of an account's runtime system prompt. CAPs modify or extend AIR behavior beyond what the Smith.ai dashboard UI can configure.

CAPs are the primary deliverable of this side of the Forward Deployment Team.

YBNYellow Box Note

The legacy name for CAPs. Older CAPs may still use "YBN" or "YELLOW BOX NOTE" in their headers. The structure and behavior are the same as CAPs; only the name evolved. New CAPs use "CAP" terminology, but recognizing "YBN" matters because it appears throughout existing artifacts and team conversation.

SPECIAL INSTRUCTIONS

The field in an account's configuration where CAP content lives. The Smith.ai system injects this field into three agents at runtime: the Main Assistant, the Intake Assistant, and the Lookup Agent. It is NOT injected into the FAQ Agent, Scheduling Agent, Service Type Agent, or Escalation Agent.

This injection scope is the single most important constraint in CAP work.

System Prompt

The full set of instructions an agent receives at runtime. The system prompt is composed of multiple sections (company info, playbook instructions, transfer logic, escalation logic, shared context, special instructions, and others) assembled by the Smith.ai platform.

Account

A Smith.ai client. Each account has its own configuration, playbooks, FAQ, services, team directory, and CAP. References to "the account" mean the specific client being worked on.

Forward Deployment TeamFDT

The umbrella team that includes both the CAP Specialists and the Solutions team. CAP Specialists work alongside the Solutions team under the FDT structure. References to "the FDT" in CAP work usually mean the CAP-focused side; references to broader cross-team coordination usually involve both groups.

02

Agents

The Smith.ai system is multi-agent. A single call may hand off between several agents. Knowing which agent is active during a given moment is essential for diagnosis.

Main Assistant

The entry-point agent that handles caller classification, FAQ checks, transfer checks, spam/sales detection, and playbook routing before handing off to the Intake Assistant.

Receives SPECIAL INSTRUCTIONS
Intake Assistant

The agent responsible for executing playbook steps: collecting caller information field-by-field, running qualification steps, and triggering end actions (transfer, end-with-message, schedule). This is the agent most CAPs target.

Receives SPECIAL INSTRUCTIONS
FAQ Agent

Handles information-seeking questions ("What are your hours?", "Where are you located?"). Invoked via startFAQSession. Has restricted knowledge sources (Company Information, FAQ list, Services list only). CAP rules cannot reach FAQ Agent behavior.

Does NOT receive SPECIAL INSTRUCTIONS
Scheduling Agent

Handles appointment scheduling, rescheduling, and cancellation. Invoked via startSchedulingSession. CAP rules cannot reach Scheduling Agent behavior.

Does NOT receive SPECIAL INSTRUCTIONS
Service Type Agent

Also called the Service Qualification Agent. Handles service-based qualification: matching the caller's stated need against the account's service list. Invoked when a playbook includes a Service-based qualification step.

Does NOT receive SPECIAL INSTRUCTIONS
Escalation Agent

Handles transfer execution. Invoked via startEscalationSession. Routes transfers to specific people, extensions, or general human-agent requests. The team directory is injected here. CAP rules cannot reach Escalation Agent behavior; they can only intercept BEFORE startEscalationSession is called or influence behavior AFTER the escalation session returns.

Does NOT receive SPECIAL INSTRUCTIONS
Lookup Assistant

Handles external data source lookups (typically backed by n8n workflows). Invoked via startLookupSession. Follows a native 3-step design: understand, lookup, done.

Receives SPECIAL INSTRUCTIONS (under a dedicated header)
03

Playbooks

The structures the Intake Assistant executes.

Playbook

A configured intake flow for a specific caller type. Accounts typically have multiple playbooks: New Client, Existing Client, All other calls, and sometimes Sales. Each playbook defines the steps the Intake Assistant must execute.

Main Playbook

A top-level playbook routed to from the Main Assistant. New Client, Existing Client, and All other calls are main playbooks.

Sub-Playbook

A playbook routed to from another playbook (typically from within the New Client playbook into a practice-area-specific flow). Sub-playbooks are loaded via getPlaybookInstructions called from inside an active playbook.

PNCPotential New Client

A caller who has not previously worked with the account. Routed to the New Client playbook. Often used in legal industry contexts.

ECExisting Client

A caller who has previously worked with the account. Routed to the Existing Client playbook.

Caller Classification

The Main Assistant's determination of caller type (PNC, EC, Other, Sales). Used to select the correct playbook. The system asks "Are you a new customer_noun or have you worked with us before?" unless a strong signal auto-classifies the caller.

Caller classification is confidential. The system enforces across all agents that EC/PNC status must NEVER be revealed to the caller. Phrases like "Since you've worked with us before" or "As a returning client" are forbidden.
customer_noun

An account-configurable noun used in caller classification phrasing, swapping in the term that matches the account's industry. Examples: "client" for a law firm, "patient" for a medical practice, "customer" for a service business.

Step

A discrete unit within a playbook. Each playbook has numbered steps that must execute in order. Steps cannot be skipped. The base playbook structure has 3 steps; UI configuration adds more.

Mandatory Field

A field the Intake Assistant is required to collect during a specific step. Until all mandatory fields for a step are collected, the step does not clear and the Intake Assistant continues asking.

Qualification Step

A step that determines whether the caller qualifies for the service. Three types: Service-based, Location-based, Custom. Qualification steps run in their own agent contexts and cannot be modified by a CAP. A CAP can override the step (intercept before it runs) or consume its result.

End Action

The terminal action of a playbook. Common end actions: endCallWithMessage, transferCall, scheduleAppointment. Determined by playbook configuration.

04

Callable actions

The functions agents can execute. Knowing the exact signature matters because CAPs script these actions verbatim.

transferCall

Executes a call transfer. Signature varies by configuration.

transferCall(destination = "[phone_number]", transfer_type = "transfer_to_pro")

Extension form adds extension = "[extension]". Requires an explicit destination phone number; cannot resolve names natively.

endCallWithMessage

Ends the call with a custom spoken message.

endCallWithMessage(message_key = "[message content]")

The message is what the AIR speaks before hanging up.

endCallWithFinalStatement

Ends the call using the playbook's configured final statement. The default end action for most playbooks.

startEscalationSession

Hands the call to the Escalation Agent for transfer execution. Passes structured context: REQUESTED, NAME, PHONE, REASON.

startFAQSession

Hands the call to the FAQ Agent for an information-seeking question. Returns to the calling agent when complete.

startSchedulingSession

Hands the call to the Scheduling Agent.

startLookupSession

Hands the call to the Lookup Agent for an external data lookup.

getPlaybookInstructions

Loads the instructions for a specific playbook. Used for sub-playbook routing. Can route to playbooks not listed in AVAILABLE PLAYBOOKS by name.

getNodeInstructions

Loads the instructions for a specific qualification step. Triggered when a qualification step begins.

onLookupSessionDone

Returns control from the Lookup Agent to the Intake Assistant.

onEscalationSessionDone

Returns control from the Escalation Agent to the calling agent (when the escalation was cancelled or resumed normally, not when a transfer executed).

performSingleExternalLookup / performMultipleExternalLookup

The Lookup Agent's actual lookup execution functions. Backed by n8n workflows in most accounts.

05

Transfers

How calls move from the AIR to humans, and the rules that govern those handoffs.

Cold Transfer

The default transfer type. The AIR transfers the caller without screening the receiving party.

Warm Transfer

A transfer where the AIR speaks with the receiving party BEFORE connecting the caller (a screening conversation). Configured per account. When enabled, two additional system prompts activate: Transfer Delegation and Warm Transfer Screening.

Transfer Delegation

The system prompt governing the handoff between the AIR and the receiving party during a warm transfer. Only active on warm-transfer-enabled accounts.

Warm Transfer Screening

The system prompt governing the screening conversation with the transfer recipient. Only active on warm-transfer-enabled accounts.

Native Team Transfers

The Smith.ai dashboard feature that auto-routes transfers based on the configured team directory. Some accounts disable this in favor of CAP-driven deterministic transfer logic.

Team Directory

The list of team members and their phone numbers/extensions. Injected only into the Escalation Agent.

Insistence

A pattern where the caller responds to a question ONLY with a transfer demand, providing zero information. Triggers immediate transfer, skipping remaining questions.

Providing any information alongside a transfer request is NOT insistence.
escalation_type

An account configuration setting that determines how the Escalation Agent handles caller-initiated human-agent requests. When set, the agent collects reason/name/phone and transfers. When None, the agent is FORBIDDEN from calling transferCall for caller-initiated requests and runs a two-turn name/phone collection instead.

06

Platform & operational terms

The infrastructure, tooling, and operational vocabulary around CAP work.

Event Log

A per-call diagnostic record showing every agent invocation, function call, parameter, and response. The primary diagnostic artifact for failure analysis, alongside transcripts.

Transcript

The text record of a call's spoken content. Used to diagnose behavioral failures.

n8n

The workflow automation platform that backs most Lookup Agent external lookups. Lookup Agent function calls (search_terms, search_column) are routed through n8n.

Smith.ai Dashboard

The configuration UI where account managers and clients set up playbooks, services, FAQ, team directory, and other account behavior. Some behavior changes belong in the dashboard, not in a CAP.

VAPTTVA Prompt Test Tool

Also called the "Prompt Test CLI." Smith.ai's internal automated end-to-end testing tool for AIR prompts and configurations. Runs simulated text-based calls against caller personas defined in YAML test suites. Supports parallel execution and includes a test generator. Runs in an engineering environment.

Platinum

The highest plan tier offered to clients. Platinum accounts receive elevated quality reviews, and CAP Specialists are responsible for reviewing high-priority Platinum call interactions and recommending CAP improvements when needed (referenced in the role overview as "Call Reviews for Platinum QA").

Version 5.2

Refers to ChatGPT 5.2, the LLM currently powering AIR. References to "5.2 standards" or "5.2 compatibility" mean writing CAPs in ways that work well with this model's behavior. As the underlying LLM evolves, CAPs may need refactoring to align with newer version standards. The role overview's "Adapting / Correcting CAPs for 5.2" responsibility refers to this work.
