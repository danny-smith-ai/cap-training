Before you use this doc

The two rules that govern all of these.

Every action in this doc follows the same two rules. Violating either one causes real problems on real calls.

Rule 1: Always write function calls exactly

Copy the syntax from this doc verbatim. The agent parses function calls at runtime. A missing quote, a wrong parameter name, or an abbreviated destination will cause the action to fail silently or misbehave. There is no partial execution — it either fires correctly or it doesn't.

Destination phone numbers must be the full E.164 format. That means +1XXXXXXXXXX — the + sign, the country code, then the full number with no spaces, dashes, or parentheses. Confirm the number with the requester before building. A wrong number in a CAP fires transfers to the wrong place, silently, on every call that hits the rule.

Rule 2: Always use "silently execute"

Every function call in a CAP should be prefaced with "silently execute" or "silently call." This instructs the agent not to narrate the function call to the caller. Without it, the agent may say something like "I'm going to transfer you now" or "Let me end this call" before executing the action — behavior most clients don't want and didn't ask for.

✓  silently execute transferCall(destination = "+1XXXXXXXXXX", transfer_type = "transfer_to_pro")
✗  execute transferCall(destination = "+1XXXXXXXXXX", transfer_type = "transfer_to_pro")
✗  transferCall(destination = "+1XXXXXXXXXX", transfer_type = "transfer_to_pro")

The single tool per response rule. An agent can only fire one function call per response turn. If your CAP has a step that collects a field AND fires a function call, those must be in separate response turns. The agent will collect the field, return to the caller, then fire the function call on the next turn. You don't need to script this separation explicitly — the agent handles it — but knowing it exists explains why you sometimes see an extra exchange in a transcript between the last collected field and the end action.

Transfer actions

Transfers.

Used to route the caller to a human or to a specific destination. Always requires an explicit phone number — the agent cannot resolve names to numbers.

transferCall Standard transfer

The most common transfer action. Routes the caller to a phone number without screening the receiving party first (cold transfer).

silently execute transferCall(destination = "+1XXXXXXXXXX",
                             transfer_type = "transfer_to_pro")
ParameterRequiredValue
destinationRequiredFull phone number in E.164 format: "+1XXXXXXXXXX"
transfer_typeRequiredAlways "transfer_to_pro" for a direct transfer
transferCall With extension

Same as above but routes to a specific extension at the destination number. Use when the client has a multi-extension phone system and wants the caller routed to a specific person or department.

silently execute transferCall(destination = "+1XXXXXXXXXX",
                             transfer_type = "transfer_to_pro",
                             extension = "207")
ParameterRequiredValue
destinationRequiredFull phone number in E.164 format
transfer_typeRequired"transfer_to_pro"
extensionRequired for extension routingExtension number as a string: "207"

Confirm the extension with the requester. Extension numbers are not validated at deploy time — a wrong extension will silently route to the wrong person.

transferCall vs startEscalationSession. transferCall is what a CAP scripts directly — it fires the transfer from the Intake Assistant immediately. startEscalationSession is the system invocation that hands control to the Escalation Agent, which then routes using the team directory. For CAP-driven transfers where you know the exact destination and want deterministic routing, always use transferCall directly. Use startEscalationSession only when you specifically want the Escalation Agent to take over (rare in CAP work).

End call actions

Ending the call.

Two options: end with a custom spoken message, or end with the playbook's configured final statement. Use the custom message when the client needs the AI to say something specific. Use the final statement for everything else.

endCallWithMessage Custom spoken message

Ends the call after the AI speaks a specific message you define in the CAP. Use this when the default end-of-call wording isn't appropriate for the scenario — disqualified callers, specific routing outcomes, scenarios where the client wants custom language.

silently execute endCallWithMessage(message_key = "Thank you for calling.
A member of our team will be in touch shortly. Have a great day.")
ParameterRequiredValue
message_keyRequiredThe exact text the AI will speak before ending the call. Wrap in double quotes. No character limit, but keep it conversational — long messages sound robotic.
endCallWithMessage Multi-language variant

When an account serves callers in more than one language, the closing message should match the language of the call. Structure the action inside a language conditional.

silently execute endCallWithMessage with this behavior:
  - If the call language is English:
    message_key = "Thank you for that information. Someone from our
    office will be in touch shortly. Have a great day."
  - If the call language is Spanish:
    message_key = "Gracias por esa información. Alguien de nuestra
    oficina se comunicará con usted en breve. Que tenga un buen día."

Always provide the Spanish message when the account has Spanish-speaking callers. An English-only endCallWithMessage will be spoken in English to a Spanish-language caller.

endCallWithFinalStatement Playbook default

Ends the call using whatever closing message the playbook has configured in the dashboard. Use this when you want the call to end normally after a CAP rule fires — you're controlling when it ends but not what it says.

silently execute endCallWithFinalStatement(playbook_id = "...")
ParameterRequiredValue
playbook_idOften omittedIn most CAPs this parameter is left as "..." or omitted entirely — the system resolves the active playbook at runtime. Only specify it if you need to force a specific playbook's final statement.

Collect all required fields before firing an end action. A CAP must not fire endCallWithMessage or transferCall until all fields required for that route have been collected. If the end action fires before collection is complete, the account loses the caller's information. State this explicitly in the CAP if the route has any ambiguity: "After collecting all of the above, silently execute…"

Agent handoff actions

Handing off to other agents.

Less commonly scripted in CAPs than transfer and end-call actions, but important to know. These hand control to a specific agent and return when that agent's session is complete.

startFAQSession Invoke FAQ Agent

Hands the call to the FAQ Agent to answer an information-seeking question, then returns control to the calling agent. Most commonly invoked automatically by Main or Intake — you'll rarely script this directly in a CAP. The more common CAP pattern for FAQ work is to house additional FAQs in the CAP itself and have Intake answer them, bypassing the FAQ Agent entirely.

silently execute startFAQSession

Remember: the FAQ Agent does not receive SPECIAL INSTRUCTIONS. If you invoke it, your CAP rules have no reach inside that session.

startSchedulingSession Invoke Scheduling Agent

Hands the call to the Scheduling Agent to book, reschedule, or cancel an appointment. Returns control when the scheduling session ends. Typically invoked automatically by Intake at the end of a playbook that has scheduling as its end action. You'll script this directly only when you need to trigger scheduling at a non-standard point in the call flow.

silently execute startSchedulingSession

Remember: the Scheduling Agent does not receive SPECIAL INSTRUCTIONS. CAP rules don't reach inside the scheduling session.

startLookupSession Invoke Lookup Agent

Hands the call to the Lookup Agent to run an external data lookup (typically via n8n). Returns control when the lookup completes. The Lookup Agent does receive SPECIAL INSTRUCTIONS under a dedicated header — CAP rules can reach it, unlike the other handoff agents.

silently execute startLookupSession

CAP instructions for the Lookup Agent are written under a separate header in the SPECIAL INSTRUCTIONS field, not mixed in with the Intake instructions. Follow the account's existing Lookup section structure when adding or modifying Lookup behavior.

Writing conventions

Defensive language and field collection.

The function call is only half the job. What you write around it determines whether the agent behaves correctly before and after the action fires.

Defensive language — stop the agent from doing the wrong thing

After a function call fires, the agent's default behavior is to continue the conversation — ask more questions, offer more help, run more intake steps. Defensive language explicitly prohibits this. Use it whenever a CAP action should terminate the flow.

silently execute transferCall(destination = "+1XXXXXXXXXX",
                             transfer_type = "transfer_to_pro")
Do not ask further questions, confirm details, or provide any information.
End the interaction after the transfer command is executed.

The three prohibition lines above are load-bearing. Without them:

  • The agent may ask for name, phone, and email before or after the transfer fires
  • The agent may confirm the transfer verbally to the caller
  • The agent may continue intake after the transfer attempt returns

Field collection gate — prevent premature action

When a route collects fields and then fires an action, use explicit gate language to prevent the action from firing before all fields are collected.

Collect the following, one by one, individually.
Do not skip any unless already collected:
  1. Caller's full name
  2. Caller's phone number
  3. Caller's email address

After collecting all of the above, silently execute
endCallWithMessage(message_key = "Thank you for that information.
Someone from our team will be in touch. Have a great day.")

"One by one, individually" — prevents the agent from asking for multiple fields in a single response turn.

"Do not skip any unless already collected" — allows the agent to skip fields the caller has already provided voluntarily, without violating the field collection rule.

"After collecting all of the above" — the gate. The action cannot fire until every field above it is collected.

Trigger precision — "identifies as" vs "mentions"

The difference between these two is significant.

✓  If the caller identifies as a member of [group]...
✗  If the caller mentions [group]...

"Mentions" fires on any reference — a caller saying "my sister called ICE on my landlord" mentions ICE. "Identifies as a member of ICE" fires only when the caller is claiming to be from ICE. Use "identifies as" for caller-type triggers. Use specific keyword lists for topic triggers.

The checklist before you paste a CAP into the dashboard

  • Every function call has "silently execute" in front of it
  • Every transferCall has a real E.164 phone number in the destination parameter
  • Every transferCall with an extension has the correct extension confirmed by the requester
  • Every endCallWithMessage has a complete, conversational message in message_key
  • Multi-language accounts have both language variants in any endCallWithMessage
  • Every route that collects fields has "after collecting all of the above" before the end action
  • Every conditional transfer has defensive language after the function call
  • Trigger phrases use "identifies as" for caller-type triggers, not "mentions"
